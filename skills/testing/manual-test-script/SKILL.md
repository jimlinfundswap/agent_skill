# 人工測試腳本 (Manual Test Script)

這個 skill 幫助您撰寫**標準化的人工測試腳本**，讓 QA、PM、業務人員都能輕鬆執行測試，確保功能品質。

## 概述

本 skill 基於以下原則：
- **商業導向**：使用業務語言描述，非技術人員也能理解
- **模擬真實用戶**：測試步驟反映真實用戶的操作行為
- **完整覆蓋**：涵蓋正向、負向、邊界三種測試類型
- **可追溯**：每個測試案例都有唯一編號，便於追蹤與回報

## 適用情境

- 🎯 **新功能驗收**：開發完成後，需要人工驗證功能是否符合規格

- 🎯 **回歸測試**：版本更新前，確保既有功能未受影響

- 🎯 **UAT 測試**：讓業務或客戶執行的使用者驗收測試

- 🎯 **跨部門協作**：讓非技術人員（PM、業務、客服）也能執行測試

---

## 測試腳本格式

### 標準表格格式

```markdown
| Test Case ID | Type | Scenario | Preconditions | Steps | Test Data | Expected Result |
| --- | --- | --- | --- | --- | --- | --- |
| {ID} | {Type} | {情境描述} | {前置條件} | {操作步驟} | {測試資料} | {預期結果} |
```

### 欄位定義

| 欄位 | 說明 | 撰寫要點 |
|------|------|---------|
| **Test Case ID** | 測試案例唯一編號 | 格式：`{專案}-{模組}-{功能}-{編號}` |
| **Type** | 測試類型 | Positive / Negative / Edge |
| **Scenario** | 測試情境 | 一句話描述「做什麼」 |
| **Preconditions** | 前置條件 | 執行測試前必須滿足的狀態 |
| **Steps** | 操作步驟 | 編號列表，描述具體操作 |
| **Test Data** | 測試資料 | 測試所需的輸入資料 |
| **Expected Result** | 預期結果 | 編號列表，描述預期的系統回應 |

---

## Test Case ID 命名規則

### 格式
```
{專案代碼}-{模組}-{功能}-{流水號}
```

### 範例
| 專案 | 模組 | 功能 | 完整 ID |
|------|------|------|---------|
| FundSwap (FS) | FUND | SEARCH | FS-FUND-SEARCH-001 |
| FundSwap (FS) | USER | LOGIN | FS-USER-LOGIN-001 |
| FundSwap (FS) | ORDER | CHECKOUT | FS-ORDER-CHECKOUT-001 |

### 編號規則
- 編號從 001 開始，依序遞增
- 同一功能的測試案例使用連續編號
- 建議依 Type 順序排列：Positive → Negative → Edge

## 測試類型 (Type) 定義

### Positive（正向測試）
驗證系統在**正常情況**下的行為是否符合預期。

**特徵**：
- 使用有效的輸入資料
- 遵循正常的操作流程
- 預期系統正確回應

**範例情境**：
- 使用正確帳號密碼登入
- 輸入有效金額進行儲值
- 選擇有效篩選條件查詢

---

### Negative（負向測試）
驗證系統在**異常情況**下的錯誤處理是否正確。

**特徵**：
- 使用無效的輸入資料
- 故意違反業務規則
- 預期系統顯示適當的錯誤訊息

**範例情境**：
- 使用錯誤密碼登入
- 輸入超過限額的金額
- 未登入狀態存取需授權功能
- 搜尋不存在的關鍵字

---

### Edge（邊界測試）
驗證系統在**邊界條件**或**特殊情況**下的行為。

**特徵**：
- 測試極端值（最大、最小、零）
- 測試特殊字元或格式
- 測試操作順序或狀態轉換
- 測試瀏覽器行為（上一頁、重新整理）

**範例情境**：
- 輸入最大允許金額
- 搜尋含特殊字元的關鍵字
- 重複快速點擊按鈕
- 使用瀏覽器上一頁返回

## 撰寫原則

### 1. 情境描述 (Scenario)

**原則**：一句話說明「測試什麼」

✅ **好的寫法**：
- 「預設進入基金搜尋頁顯示基金清單」
- 「使用關鍵字搜尋基金名稱」
- 「未登入狀態勾選『我的關注』」

❌ **不好的寫法**：
- 「測試搜尋功能」（太模糊）
- 「驗證 API 回傳正確」（技術導向）

---

### 2. 前置條件 (Preconditions)

**原則**：說明執行測試前必須滿足的狀態

✅ **好的寫法**：
- 「已進入基金搜尋頁」
- 「使用者已登入」
- 「購物車中有至少一項商品」
- 「無」（若無特殊前置條件）

❌ **不好的寫法**：
- 「資料庫已有測試資料」（技術導向）
- 「API 服務正常」（技術導向）

---

### 3. 操作步驟 (Steps)

**原則**：
- 使用編號列表
- 每步驟描述一個具體操作
- 使用使用者視角的動詞
- 步驟間有明確的順序關係

✅ **好的寫法**：
```
1. 從首頁導覽列點擊「找基金 > 基金搜尋」
2. 在搜尋框輸入關鍵字「摩根」
3. 按下查詢按鈕
4. 檢視搜尋結果清單
```

❌ **不好的寫法**：
```
1. 呼叫 /api/funds/search
2. 傳入 keyword=摩根
```

**常用動詞**：
| 動作類型 | 建議用詞 |
|---------|---------|
| 導覽 | 點擊、前往、開啟、進入 |
| 輸入 | 輸入、填寫、選擇、勾選 |
| 確認 | 按下、點擊、確認、送出 |
| 檢視 | 檢視、確認、觀察、檢查 |
| 等待 | 等待、等候載入完成 |

---

### 4. 測試資料 (Test Data)

**原則**：
- 列出測試所需的輸入值
- 使用具體的範例資料
- 敏感資料使用替代值

✅ **好的寫法**：
```
關鍵字：摩根
篩選條件：境外基金 + 有配息
測試帳號：test@example.com
```

❌ **不好的寫法**：
```
使用有效資料
```

---

### 5. 預期結果 (Expected Result)

**原則**：
- 使用編號列表
- 描述可觀察、可驗證的結果
- 涵蓋 UI 顯示、資料內容、系統行為

✅ **好的寫法**：
```
1. 頁面網址為 `/trade/funds/`
2. 頁面標題顯示「找基金」與總筆數
3. 清單顯示符合條件的基金卡片
4. 每張卡片包含基金名稱、風險等級、績效
5. 頁面底部顯示分頁控制元件
```

❌ **不好的寫法**：
```
1. 功能正常
2. 資料正確
```

## 常見測試情境模板

### 搜尋功能

| Type | Scenario 模板 |
|------|--------------|
| Positive | 使用關鍵字搜尋{對象}名稱 |
| Positive | 使用部分關鍵字搜尋（模糊搜尋） |
| Negative | 輸入不存在的關鍵字搜尋 |
| Edge | 搜尋含特殊字元的關鍵字 |
| Edge | 搜尋空白或只有空格 |

---

### 篩選功能

| Type | Scenario 模板 |
|------|--------------|
| Positive | 篩選只顯示「{條件}」 |
| Positive | 套用多重篩選條件 |
| Negative | 套用篩選條件後查無任何結果 |
| Edge | 套用篩選後再取消篩選 |
| Edge | 清空所有篩選標籤 |

---

### 排序功能

| Type | Scenario 模板 |
|------|--------------|
| Positive | 切換排序為「依{欄位} 高→低」 |
| Positive | 切換排序為「依{欄位} 低→高」 |
| Edge | 重複切換排序方式 |

---

### 分頁功能

| Type | Scenario 模板 |
|------|--------------|
| Positive | 使用分頁功能切換至第二頁 |
| Edge | 前往最後一頁再點擊下一頁 |
| Edge | 直接以網址參數開啟特定頁碼 |

---

### 登入相關

| Type | Scenario 模板 |
|------|--------------|
| Positive | 使用正確帳號密碼登入 |
| Negative | 使用錯誤密碼登入 |
| Negative | 未登入狀態存取需授權功能 |
| Edge | 登入後使用瀏覽器上一頁 |

---

### 表單送出

| Type | Scenario 模板 |
|------|--------------|
| Positive | 填寫完整資料送出表單 |
| Negative | 必填欄位留空送出 |
| Negative | 輸入超過欄位限制的內容 |
| Edge | 快速重複點擊送出按鈕 |

## 完整範例

### 範例：基金搜尋功能

```markdown
# FundSwap 找基金：基金搜尋 人工測試腳本

| Test Case ID | Type | Scenario | Preconditions | Steps | Test Data | Expected Result |
| --- | --- | --- | --- | --- | --- | --- |
| FS-FUND-SEARCH-001 | Positive | 預設進入基金搜尋頁顯示基金清單 | 無 | 1. 從首頁導覽列點擊「找基金 > 基金搜尋」 2. 確認頁面載入完成 3. 檢視頁面顯示的基金清單 | 無 | 1. 頁面網址為 `/trade/funds/` 2. 頁面標題顯示「找基金」與總筆數 3. 顯示基金清單卡片 4. 頁面底部顯示分頁控制元件 |
| FS-FUND-SEARCH-002 | Positive | 使用關鍵字搜尋基金名稱 | 已進入基金搜尋頁 | 1. 在搜尋框輸入關鍵字 2. 按下查詢 3. 檢視搜尋結果 | 關鍵字：摩根 | 1. 清單更新為符合關鍵字的基金 2. 每筆基金名稱包含「摩根」 3. 總筆數更新 |
| FS-FUND-SEARCH-003 | Negative | 輸入不存在的關鍵字搜尋 | 已進入基金搜尋頁 | 1. 在搜尋框輸入不存在的關鍵字 2. 按下查詢 3. 檢視搜尋結果 | 關鍵字：xyzabc123 | 1. 顯示「查無搜尋結果」提示 2. 總筆數顯示為 0 3. 不顯示任何基金卡片 |
| FS-FUND-SEARCH-004 | Edge | 搜尋含特殊字元的關鍵字 | 已進入基金搜尋頁 | 1. 在搜尋框輸入含特殊字元的關鍵字 2. 按下查詢 3. 檢視搜尋結果 | 關鍵字：S&P | 1. 系統正常處理特殊字元 2. 顯示符合條件的基金或「查無搜尋結果」 3. 不會出現系統錯誤 |
```

## 使用方式

### 步驟 1：確認測試範圍

從規格文件或 PRD 中識別需要測試的功能：
- 主要功能流程
- 業務規則與驗證
- 使用者介面互動

### 步驟 2：規劃測試案例

依據功能特性，規劃三種類型的測試：

| 順序 | Type | 目的 |
|------|------|------|
| 1 | Positive | 驗證正常流程 |
| 2 | Negative | 驗證錯誤處理 |
| 3 | Edge | 驗證邊界情況 |

### 步驟 3：撰寫測試腳本

1. 建立 Markdown 檔案，標題格式：`{專案名稱}：{功能名稱} 人工測試腳本`
2. 使用標準表格格式撰寫測試案例
3. 依 Test Case ID 順序排列

### 步驟 4：審閱與執行

- [ ] 測試案例涵蓋所有主要功能
- [ ] 步驟描述清楚，非技術人員可理解
- [ ] 預期結果具體可驗證
- [ ] 測試資料已準備完成

---

## 注意事項

- ⚠️ **避免技術術語**：使用業務語言，讓 PM、業務都能執行測試

- ⚠️ **步驟要具體**：避免「操作功能」這類模糊描述

- ⚠️ **結果要可驗證**：避免「功能正常」這類無法驗證的描述

- 💡 **模擬真實用戶**：想像用戶實際操作的情境與行為

- 💡 **考慮邊界情況**：特殊字元、極端值、重複操作、瀏覽器行為

- 💡 **保持一致性**：同一專案使用相同的命名規則與格式
